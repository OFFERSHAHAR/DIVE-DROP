Self‑Healing n8n Workflows

איך בונים מערכת שבה Workflow שנופל ב‑n8n שולח את השגיאה למחשב המקומי, Claude Code מנתח, מתקן, ומחזיר את התיקון ל‑n8n Cloud.

n8n CloudClaude CodeMCPngrokBridge Server
הרעיון ב‑30 שניות

מה המערכת עושה?

כש‑Workflow נכשל
n8n מפעיל Error Trigger ושולח את פרטי התקלה החוצה.
ה‑Bridge המקומי מקבל
שרת קטן במחשב שלך מקבל Webhook דרך ngrok.
Claude Code נכנס לזירה
הוא מושך את ה‑Workflow דרך MCP, מבין את הבעיה ומכין תיקון.
התיקון חוזר ל‑n8n
Claude משתמש ב‑API/MCP כדי לעדכן את ה‑Workflow בענן.

התוצאה: פחות כיבוי שריפות ידני, יותר “המכונה מתקנת את עצמה”.

מה צריך לפני שמתחילים?

n8n Cloud

חשבון פעיל ו‑API Key מתוך Settings → API.

Node.js

גרסה 18 ומעלה כדי להריץ שרת Express וכלי CLI.

Claude Code

CLI של Anthropic, מחובר לחשבון עם API key וקרדיטים.

ngrok

מנהרה מאובטחת מהאינטרנט למחשב המקומי.

MCP Server

מחבר בין Claude לבין n8n כדי לקרוא ולעדכן Workflows.

סוד גישה

BRIDGE_API_KEY כדי שרק n8n שלך יוכל לקרוא לשרת.

שלב 1

התקנת Claude Code

npm install -g @anthropic-ai/claude-code claude claude --version

פותחים PowerShell / Terminal, מתקינים, מתחברים לחשבון Anthropic, ואז בודקים שהפקודה זמינה.

שלב 2

חיבור n8n MCP ל‑Claude

בקובץ ההגדרות של Claude מוסיפים MCP Server שמצביע ל‑n8n Cloud שלך.

{ "mcpServers": { "n8n-mcp": { "command": "npx", "args": ["-y", "@czlonkowski/n8n-mcp@latest"], "env": { "N8N_HOST": "https://your-instance.app.n8n.cloud", "N8N_API_KEY": "your-n8n-api-key-here" } } } }

חשוב: להחליף את ה‑URL והמפתח בפרטים האמיתיים שלך.

שלב 3

Bridge Server מקומי

יוצרים שרת Express קטן שמאזין ל‑POST בנתיב /fix-workflow, בודק API key, בונה Prompt עם פרטי השגיאה, ומפעיל Claude Code.

mkdir claude-n8n-bridge cd claude-n8n-bridge npm init -y npm install express node server.js
Endpoint חשוב: http://localhost:3456/health מחזיר status ok כדי לבדוק שהשרת חי.
שלב 4

פתיחה לאינטרנט עם ngrok

ngrok נותן URL ציבורי בסגנון:

https://abc123xyz.ngrok-free.app -> http://localhost:3456

בגרסה החינמית ה‑URL משתנה בכל הפעלה מחדש. צריך לעדכן אותו ב‑n8n.

בונים Workflow שנשבר בכוונה

מטרה

לוודא שכל השרשרת עובדת: n8n → Error Handler → ngrok → Bridge → Claude → תיקון.

המשימה

  1. צור Workflow בשם TEST - Intentional Bug.
  2. הוסף Manual Trigger.
  3. הוסף Code node עם הבאג למטה.
  4. חבר Error Workflow ל‑Claude Code Fixer.
  5. הרץ Test Workflow וצפה בטרמינל.

קוד שבור

const data = $input.all(); const userName = data[0].json.user.profile.name; return [{ json: { name: userName } }];

const data = $input.all(); const userName = data[0]?.json?.user?.profile?.name ?? 'Unknown'; return [{ json: { name: userName } }];

בדיקת עומק קצרה

1. למה צריך Bridge?

כי n8n Cloud לא יכול להריץ Claude Code על המחשב שלך ישירות. ה‑Bridge הוא המתורגמן והדלת.

2. למה צריך ngrok?

כי המחשב המקומי לא חשוף לאינטרנט. ngrok יוצר כתובת HTTPS ציבורית.

3. למה צריך MCP?

כדי ש‑Claude יוכל לבצע פעולות n8n אמיתיות: לקרוא Workflow, לעדכן ולוודא.

4. מה הסיכון?

Auto‑fix על Workflows קריטיים עלול לשנות לוגיקה לא נכון. מתחילים בסביבה בטוחה.

סיכום מנהלים

מה למדנו?

  • Self‑Healing הוא לא קסם, אלא שרשרת Webhook + Bridge + Agent + API.
  • n8n מזהה נפילה, Claude Code מנתח, MCP נותן לו ידיים בתוך n8n.
  • המערכת טובה לתיקוני קוד/Expressions/Null checks, לא לתקלות הרשאות, Billing או API חיצוני שנפל.
  • לפני Production: בדיקות, גיבויים, הגבלת הרשאות, ולוגים לכל תיקון.
משפט זהב: אל תיתן לרובוט מפתח לכל הבית לפני שראית אותו מסדר מגירה אחת.
